# Java와 JVM

## `volatile`, `synchronized`, atomic은 어떻게 구분하는가?

`volatile`은 변수의 변경을 다른 스레드가 볼 수 있게 하는 가시성을 보장한다.
`count++`처럼 읽기-수정-쓰기 단계가 있는 연산의 원자성은 보장하지 않는다.

`synchronized`는 임계 영역의 원자성과 가시성을 함께 보장한다.
여러 상태를 하나의 규칙으로 변경해야 할 때 명확한 선택이다.

`AtomicInteger` 같은 atomic 계열은 CAS를 사용한다.
단순 카운터에는 유리하지만, 여러 연산을 하나로 묶는 업무 규칙까지 원자적으로 만들지는 않는다.

## CAS의 ABA 문제와 경합 비용은 무엇인가?

CAS는 값이 예상값과 같을 때만 변경한다.
값이 `A → B → A`로 바뀌면 처음부터 `A`였다고 오인할 수 있는데 이를 ABA 문제라고 한다.

상태 전이가 중요한 경우에는 버전 번호를 함께 비교하는 `AtomicStampedReference`를 검토한다.
또한 경합이 심하면 CAS 재시도가 CPU를 소모할 수 있으므로, 항상 락보다 빠르다고 가정하면 안 된다.

## 불변 객체와 동시성 컬렉션만으로 스레드 안전성을 보장할 수 있는가?

완전히 보장하지는 않는다.
불변 객체와 `final` 필드는 안전한 공개를 쉽게 만들고, 동시성 컬렉션은 개별 연산을 안전하게 만든다.

하지만 "재고가 0보다 클 때만 차감"처럼 여러 연산을 묶은 규칙에는 별도의 원자성 보장이 필요하다.
동기화, DB 조건부 갱신, 메시지 직렬화 등 업무 경계에 맞는 제어를 선택한다.

## GC 문제는 어떻게 진단하는가?

Full GC 횟수만으로 원인을 단정하지 않는다.
GC 로그에서 pause 시간, heap 사용량 변화, allocation rate, promotion, old generation 점유율을 본다.

동시에 요청 지연, CPU, 스레드 수, DB 대기 시간을 대조한다.
객체 생성량이 많은 경로, 큰 컬렉션, 캐시 누수, 커넥션·스레드 누수처럼 원인을 분리해 확인한다.

## Virtual Thread는 언제 유리하고 무엇을 해결하지 못하는가?

Virtual Thread는 I/O 대기가 많은 요청을 적은 플랫폼 스레드로 처리하는 데 유리하다.
하지만 DB 커넥션, 외부 API 동시 호출, CPU는 무한히 늘지 않는다.

따라서 커넥션 풀 크기, 외부 호출 timeout, rate limit, CPU bound 작업의 executor는 여전히 제한해야 한다.
동기화 블록에서 오래 대기하거나 pinning이 발생하는 경로도 점검 대상이다.

# Spring Boot와 JPA

## `@Transactional`이 적용되지 않는 경우는 무엇인가?

Spring의 기본 트랜잭션은 프록시 기반 AOP다.
같은 클래스 내부 호출(self-invocation)은 프록시를 거치지 않아 적용되지 않는다.

`private` 메서드, 프록시 대상이 아닌 객체, 잘못된 예외 처리도 의도한 롤백을 막을 수 있다.
기본적으로 `RuntimeException`과 `Error`가 롤백 대상이며, checked exception은 `rollbackFor`를 명시해야 한다.

## JPA의 N+1은 어떤 방법으로 해결하며, Fetch Join은 언제 위험한가?

항상 필요한 연관 데이터는 Fetch Join이나 EntityGraph로 명시적으로 가져온다.
여러 연관 엔티티를 묶어 조회해야 하면 batch size도 선택지다.

다만 1:N 컬렉션 Fetch Join은 row를 증가시키고 페이징과 결합하면 메모리 페이징 문제가 생길 수 있다.
목록 조회에서는 DTO 직접 조회나 batch size가 더 안전한 경우가 많다.

## JPQL 벌크 연산 뒤 영속성 컨텍스트는 왜 주의해야 하는가?

벌크 `UPDATE`·`DELETE`는 영속성 컨텍스트를 거치지 않고 DB에 직접 반영된다.
이미 1차 캐시에 올라온 엔티티와 DB 값이 달라질 수 있다.

벌크 연산 후에는 `flush`와 `clear`를 적절히 수행하거나, 해당 컨텍스트의 엔티티를 다시 조회해 일관성을 맞춘다.

## `readOnly = true`는 쓰기를 막는가?

절대적인 쓰기 차단 옵션은 아니다.
JPA 구현체에서는 dirty checking과 flush를 최적화하는 힌트로 동작할 수 있고, DB·드라이버별 읽기 전용 최적화가 적용될 수 있다.

쓰기 방지는 권한, API 설계, 트랜잭션 경계, DB 권한처럼 별도의 계층에서 보장해야 한다.

## Spring Boot 자동 설정은 왜 예상과 다르게 동작할 수 있는가?

자동 설정은 classpath, 설정값, `@Conditional` 조건을 기반으로 빈을 등록한다.
의존성 하나가 추가되거나 사용자 정의 빈이 등록되면 자동 설정 결과가 달라질 수 있다.

문제 발생 시 Condition Evaluation Report, Actuator의 beans·configprops, 실제 주입된 빈을 확인한다.
자동 설정을 무조건 신뢰하기보다 활성 조건과 override 범위를 파악해야 한다.

# 데이터베이스와 트랜잭션

## 복합 인덱스 `(A, B, C, D)`는 어떻게 정렬되는가?

복합 인덱스의 B+Tree 리프 노드는 각 컬럼별로 따로 정렬되지 않는다.
`(A, B, C, D)`라는 하나의 복합 키를 사전식으로 정렬한다.

- 먼저 `A`가 정렬된다.
- 같은 `A` 안에서 `B`가 정렬된다.
- 같은 `A`, `B` 안에서 `C`, `D`가 정렬된다.

따라서 `A = ? AND B = ? AND C >= ?`는 연속된 인덱스 범위를 탐색할 수 있다.
반면 `B = ?`만으로는 리프 노드에서 연속된 범위를 바로 특정하기 어렵다.
이 특성이 복합 인덱스의 왼쪽 접두사(Leftmost Prefix) 원칙의 근거다.

## 범위 조건 뒤의 컬럼은 왜 효율이 떨어질 수 있는가?

`(A, B, C)` 인덱스와 `A = ? AND B >= ? AND C = ?` 조건을 가정한다.
`A`는 등치 조건으로 범위를 좁히지만, `B`부터는 여러 값을 읽어야 한다.
그 이후의 `C`는 인덱스 필터링에 활용될 수 있어도 탐색 범위를 강하게 줄이는 키로 쓰기 어렵다.

인덱스 순서는 보통 다음을 함께 고려한다.

- 자주 쓰는 등치 조건 컬럼
- 그 다음의 범위 조건 컬럼
- 정렬과 페이징에 필요한 컬럼
- 선택도와 실제 쿼리 빈도

`EXPLAIN ANALYZE`로 예상 행 수와 실제 행 수, 인덱스 사용 여부, 정렬 발생 여부를 확인해야 한다.

## 인덱스를 추가한 뒤 쓰기 성능은 어떻게 판단하는가?

인덱스는 조회만 빠르게 하는 기능이 아니다.
`INSERT`, 인덱스 키가 바뀌는 `UPDATE`, `DELETE`마다 인덱스 구조도 갱신해야 한다.
페이지가 가득 차면 페이지 분할이 일어나고, 무작위 키 삽입은 분할과 I/O 비용을 키울 수 있다.

측정값이 없다면 특정 지연 시간을 주장하지 않는다.
대신 다음 조건으로 재현 가능한 비교를 준비한다.

- 같은 데이터 규모와 데이터 분포
- 같은 동시성, 캐시 상태, 요청 파라미터 분포
- 인덱스 적용 전후의 API·SQL p50, p95, p99
- TPS, 오류율, CPU, 디스크 I/O, 인덱스 크기
- `INSERT`, `UPDATE` 지연과 느린 쿼리 로그

조회 개선 효과와 쓰기 비용을 함께 보고 인덱스를 유지하거나 제거한다.

## 낙관적 락과 비관적 락은 어떻게 선택하는가?

충돌이 드물면 버전 컬럼 기반 낙관적 락이 적합하다.
락을 오래 잡지 않아 동시성이 좋지만, 충돌 시 재시도 비용이 든다.

충돌이 빈번하고 작업을 직렬화해야 하면 `SELECT ... FOR UPDATE` 같은 비관적 락이나 조건부 `UPDATE`가 단순할 수 있다.
예를 들어 재고 차감은 다음처럼 조건부 갱신으로 음수 재고를 방지할 수 있다.

```sql
UPDATE stock
SET qty = qty - 1
WHERE id = :id AND qty > 0;
```

영향 행 수가 0이면 재고가 없거나 동시 변경이 있었다는 뜻으로 처리한다.
핵심은 락의 이름이 아니라 충돌 빈도, 재시도 비용, 반드시 지켜야 하는 업무 규칙이다.

## `REQUIRES_NEW`는 언제 주의해야 하는가?

`REQUIRES_NEW`는 기존 트랜잭션을 잠시 보류하고 새 물리 트랜잭션을 시작한다.
새 DB 커넥션이 필요할 수 있으므로, 외부 트랜잭션이 커넥션을 점유한 상태에서 반복 호출하면 커넥션 풀이 고갈될 수 있다.

독립 커밋이 반드시 필요한 경우에만 사용한다.
예를 들어 감사 로그처럼 본 작업의 롤백과 분리할 이유가 명확한 작업이 대상이다.
단순히 예외를 분리하려는 목적으로 사용하면 트랜잭션 경계와 장애 복구가 더 복잡해질 수 있다.

# 캐시와 Redis 분산 락

## 단일 Redis 락은 어디까지 원자적인가?

`SET key token NX PX ttl`은 단일 Redis 인스턴스에서 락 획득과 TTL 설정을 원자적으로 처리한다.
`token`은 예측하기 어려운 고유값을 사용하고, 해제할 때는 토큰이 일치할 경우에만 Lua Script로 삭제해야 한다.

단순 `DEL`은 위험하다.
락이 만료된 뒤 다른 요청이 같은 키를 획득하면, 이전 요청이 새 요청의 락을 삭제할 수 있기 때문이다.

## Redis 장애가 나면 락의 상호 배제는 보장되는가?

완전히 보장되지 않는다.
Primary-Replica 복제는 보통 비동기이므로 락이 Replica에 복제되기 전 Primary가 장애 나면, 승격된 Replica는 기존 락을 모를 수 있다.
그 결과 다른 요청이 같은 자원에 대한 락을 얻을 가능성이 있다.

따라서 Redis 락은 캐시 재생성처럼 중복 실행을 복구할 수 있는 작업에 적합하다.
재고, 결제, 잔액처럼 중복 실행 자체가 치명적인 상태 변경은 DB 트랜잭션, 조건부 갱신, 비관적 락 같은 최종 무결성 장치를 둬야 한다.

## TTL 만료 뒤 이전 요청이 늦게 도착하면 어떻게 대응하는가?

작업 시간이 TTL보다 길거나 애플리케이션이 긴 GC pause를 겪으면, 락이 만료된 뒤 다른 요청이 진입할 수 있다.
이전 보유자의 요청이 늦게 도착하면 최신 상태를 덮어쓸 위험이 있다.

이 문제에는 fencing token을 사용할 수 있다.

- 락 획득 시 단조 증가 번호를 함께 발급한다.
- 보호 대상 저장소는 마지막으로 처리한 번호보다 작은 토큰의 쓰기를 거부한다.
- 이전 보유자의 늦은 요청은 저장소 단계에서 차단된다.

대상 저장소가 토큰 비교를 지원해야 하며, Redis 락만 추가한다고 자동으로 보장되지는 않는다.

## Redlock은 어떤 조건에서 고려하는가?

Redlock은 독립 Redis 노드의 과반수에서 제한 시간 안에 락을 얻었을 때 성공으로 판단한다.
실패했거나 획득 시간이 TTL을 넘으면 이미 얻은 락을 해제한다.

노드별 짧은 timeout, 재시도 제한, 지수 백오프, jitter가 필요하다.
그러나 네트워크 분할, 지연된 패킷, 긴 GC pause가 만드는 모든 위험을 제거하지는 못한다.
강한 정합성이 필요한 작업에는 DB 락, ZooKeeper, etcd, fencing token 같은 대안을 우선 검토한다.

## 캐시 장애가 DB에 전파되는 것을 어떻게 막는가?

캐시 미스가 급증하면 DB로 요청이 몰릴 수 있다.
인기 키의 동시 만료는 Cache Stampede, 캐시 서버 장애는 Cache Avalanche로 이어질 수 있다.

- TTL에 jitter를 넣어 만료 시점을 분산한다.
- 인기 키는 단일 요청만 재계산하고 나머지는 기존 값 또는 제한된 fallback을 사용한다.
- 캐시 미스 시 rate limit, circuit breaker, 요청 병합으로 DB를 보호한다.
- 존재하지 않는 키의 반복 조회는 빈 값 캐시나 Bloom Filter로 줄인다.

캐시의 정합성 수준과 장애 시 허용 가능한 오래된 데이터의 범위를 먼저 정해야 한다.

# 멱등성과 일관성

## HTTP 메서드의 멱등성과 업무 멱등성은 무엇이 다른가?

HTTP 관점에서 `PUT`, `DELETE`는 같은 요청을 여러 번 보내도 최종 자원 상태가 같으므로 멱등적이다.
하지만 API가 실제로 어떤 부수 효과를 만들었는지는 별개의 문제다.

예를 들어 `DELETE` 요청이 삭제 알림을 매번 발송하면 자원 상태는 같아도 업무 결과는 중복될 수 있다.
중복 실행이 위험한 명령은 메서드 이름과 관계없이 별도 멱등성 설계가 필요하다.

## 멱등성 키는 어떻게 저장하고 경쟁 상태를 막는가?

멱등성 키만 메모리에 저장하면 서버 재시작이나 다중 인스턴스에서 보장할 수 없다.
공유 DB 또는 신뢰할 수 있는 저장소에 키, 요청 본문 해시, 처리 상태, 최종 응답을 저장한다.

동일 키가 동시에 들어오면 유니크 제약이나 원자적 insert로 하나만 선점한다.
나머지 요청은 완료된 결과를 반환하거나, 처리 중 상태에 맞는 응답을 반환한다.
같은 키에 본문 해시가 다르면 클라이언트 오류로 거절해야 한다.

## 멱등성 키의 만료 시간은 어떻게 정하는가?

만료 시간은 단순히 짧게 잡지 않는다.
클라이언트 재시도 가능 기간, 결제·주문의 지연 도착 가능성, 감사와 분쟁 대응 요구를 함께 고려한다.

만료 뒤 동일 키가 재사용될 수 있다는 정책을 문서화해야 한다.
중요한 업무는 주문 번호나 결제 요청 ID처럼 도메인 자체의 유니크 키를 추가로 둬야 한다.

## DB 유니크 제약은 멱등성에 어떻게 활용하는가?

애플리케이션에서 먼저 조회하고 없으면 insert하는 방식은 동시 요청에서 경쟁 상태가 생긴다.
DB의 유니크 제약으로 최종 중복 방어선을 두고, 중복 키 예외를 기존 결과 조회로 전환할 수 있다.

단, 유니크 제약은 중복 레코드 방지 수단이다.
외부 API 호출, 알림 발송, 포인트 지급 같은 부수 효과까지 되돌리거나 한 번만 실행하게 만들지는 않는다.

## 메시지 소비자의 멱등성은 어떻게 구현하는가?

At-Least-Once 전달에서는 같은 메시지가 다시 올 수 있다고 가정한다.
메시지의 고유 ID를 처리 이력 테이블에 저장하고, 이미 처리된 메시지는 건너뛸 수 있다.

업무 데이터에 자연스러운 유니크 키가 있다면 upsert나 조건부 갱신도 유용하다.
처리 이력 저장과 업무 상태 변경은 같은 로컬 트랜잭션으로 묶어야, 한쪽만 반영되는 문제를 줄일 수 있다.

## Transactional Outbox는 어떤 문제를 해결하고 무엇을 보장하지 않는가?

DB 변경과 메시지 발행을 각각 수행하면, DB만 커밋되거나 메시지만 발행되는 이중 쓰기 문제가 생긴다.
Outbox는 업무 데이터 변경과 이벤트 레코드 저장을 하나의 DB 트랜잭션으로 처리한다.

별도 릴레이가 Outbox를 읽어 브로커로 발행하므로 DB 커밋 뒤 이벤트 유실 위험을 줄인다.
하지만 릴레이 장애 후 같은 이벤트가 다시 발행될 수 있으므로 소비자 멱등성은 여전히 필요하다.

## Exactly-once는 어디까지 가능한가?

Kafka의 멱등 프로듀서와 트랜잭션은 Kafka 내부의 consume-process-produce 흐름에서 중복을 줄일 수 있다.
외부 DB, 외부 결제 API, 이메일 발송까지 포함한 전 구간의 정확히 한 번을 자동으로 보장하지는 않는다.

외부 부수 효과에는 멱등성 키, 처리 이력, Outbox, 보상 절차를 조합한다.
"Exactly-once"라는 표현을 사용할 때는 보장 범위를 명확히 해야 한다.

# 분산 처리와 메시징

## timeout 뒤 재시도된 요청은 왜 멱등해야 하는가?

timeout은 서버 작업 실패를 뜻하지 않는다.
서버는 작업을 완료했지만 응답만 유실됐을 수 있으므로, 클라이언트가 재시도하면 생성·결제 같은 명령이 중복 실행될 수 있다.

멱등성 키를 사용하면 같은 요청을 식별할 수 있다.

- 키, 요청 본문 해시, 처리 상태, 응답 결과를 저장한다.
- 같은 키와 같은 본문이면 기존 결과를 반환한다.
- 같은 키에 다른 본문이 오면 오류로 처리한다.
- 키의 보관 기간과 동시 요청 선점 방식을 정한다.

HTTP 메서드의 멱등성과 업무 명령의 멱등성은 별개다.

## Kafka에서 offset 커밋 순서는 왜 중요한가?

DB 저장 후 offset을 커밋하면, 커밋 전 장애 때 메시지가 재전달될 수 있다.
따라서 소비자는 중복 처리돼도 결과가 같은 멱등성을 가져야 한다.

반대로 offset을 먼저 커밋하면 처리 전 장애에서 메시지가 유실될 수 있다.
외부 DB나 API까지 포함한 정확히 한 번 처리는 Kafka 설정만으로 해결되지 않는다.
메시지 ID 처리 이력, upsert, outbox 같은 별도 설계가 필요하다.

## 재처리와 DLQ는 어떻게 설계하는가?

실패 메시지를 원본 토픽에서 무작정 다시 읽으면 순서와 중복 영향이 커질 수 있다.
재처리 전에 실패 원인, 대상 범위, 순서 영향, 멱등성, 복구 완료 기준을 정한다.

DLQ에는 원본 메시지뿐 아니라 실패 사유, 재시도 횟수, 발생 시각, 원본 topic·partition·offset을 남긴다.
복구는 별도 consumer group 또는 재처리 토픽을 사용하고, 결과를 관찰한 뒤 정상 흐름으로 반영한다.

## Saga 보상은 DB rollback과 무엇이 다른가?

Saga의 보상은 이미 완료된 일을 새 작업으로 상쇄하는 것이다.
시간을 되돌리는 DB rollback과 다르며, 외부 결제 취소나 이미 발송된 알림처럼 완전히 되돌릴 수 없는 작업도 있다.

각 단계에는 멱등성, 재시도, 수동 복구, 중간 상태 노출 정책이 필요하다.
설계 시에는 성공 흐름뿐 아니라 보상 실패와 반복 실행까지 정의해야 한다.

# Spring과 API

## JWT만으로 강제 로그아웃과 권한 변경을 즉시 반영할 수 있는가?

JWT는 서명 검증만으로 사용자를 식별할 수 있지만, 이미 발급된 토큰을 서버가 즉시 무효화하기는 어렵다.
Access Token은 짧게 유지하고 Refresh Token은 서버 측 저장소에서 회전, 폐기, 재사용 탐지를 관리한다.

즉시 반영이 필요하면 토큰 버전, deny-list, 세션 저장소 같은 상태 확인 수단을 추가한다.
검증 시에는 서명뿐 아니라 `exp`, `iss`, `aud`, 허용 알고리즘을 확인하고, 토큰이 지정한 알고리즘을 그대로 신뢰하지 않는다.

## 쿠키 인증과 토큰 저장에서 무엇을 구분해야 하는가?

`HttpOnly` 쿠키는 JavaScript 접근을 막아 XSS로 인한 토큰 탈취 노출을 줄인다.
대신 브라우저가 자동 전송하므로 CSRF 방어가 필요하다.

CSRF 방어에는 `SameSite`, CSRF 토큰, Origin·Referer 검증을 조합할 수 있다.
CORS는 브라우저의 교차 출처 읽기 제한일 뿐 인증·인가를 대체하지 않는다.

## 동시 수정으로 인한 Lost Update는 어떻게 막는가?

같은 자원을 여러 사용자가 수정하면 마지막 요청이 이전 변경을 덮어쓸 수 있다.
응답에 버전 또는 `ETag`를 제공하고, 클라이언트가 `If-Match`로 기대 버전을 보내게 할 수 있다.

버전이 다르면 서버는 `412 Precondition Failed` 또는 업무 의미에 맞는 `409 Conflict`로 거절한다.
DB의 낙관적 락 version 컬럼과 연결하면 API부터 저장소까지 같은 동시성 규칙을 유지할 수 있다.

# 결제와 금전 처리

## 결제 요청의 상태는 왜 `성공`과 `실패`만으로 부족한가?

결제 요청 뒤 클라이언트 연결이 끊기거나 PG 응답이 timeout 되면, 실제 승인 여부를 즉시 알 수 없을 수 있다.
이 상태를 실패로 단정하고 재시도하면 중복 결제가 발생할 수 있다.

`READY`, `PENDING`, `APPROVED`, `FAILED`, `CANCELED`처럼 중간 상태를 명시한다.
응답을 받지 못한 요청은 조회 API, PG 승인 조회, webhook으로 최종 상태를 확인한 뒤 전이한다.

## 중복 결제 요청은 어떻게 막는가?

결제 시도마다 서버가 생성한 고유 결제 ID와 클라이언트의 멱등성 키를 연결한다.
동일 요청은 기존 결제 시도와 결과를 반환하고, 다른 요청이 같은 주문을 다시 결제하지 못하도록 주문·결제 상태를 원자적으로 확인한다.

DB 유니크 제약과 조건부 상태 갱신으로 최종 방어선을 둔다.
Redis 락은 경합 완화에 사용할 수 있지만, 금전 정합성의 유일한 보장 수단이 되어서는 안 된다.

## PG 승인 요청과 DB 저장을 하나의 트랜잭션으로 묶을 수 있는가?

일반적으로 외부 PG 호출과 로컬 DB 트랜잭션을 하나의 ACID 트랜잭션으로 묶을 수 없다.
DB 트랜잭션을 오래 열고 외부 응답을 기다리면 락과 커넥션 점유 시간이 늘어난다.

결제 시도를 먼저 기록하고, PG 승인 결과를 상태 전이로 반영하는 방식이 일반적이다.
결과 반영 실패나 timeout은 승인 조회·webhook·재처리 작업으로 복구하며, 모든 상태 전이는 멱등하게 구현한다.

## webhook은 왜 검증과 멱등 처리가 필요한가?

webhook은 네트워크 장애로 재전달될 수 있고, 순서가 바뀌거나 사용자가 아닌 외부에서 위조될 수도 있다.
PG가 제공하는 서명, timestamp, 이벤트 ID를 검증하고 허용된 발신지 정책을 적용한다.

이벤트 ID를 처리 이력으로 저장해 중복을 제거한다.
현재 결제 상태에서 허용되는 전이만 적용해, 늦게 도착한 과거 이벤트가 최신 상태를 되돌리지 못하게 한다.

## 결제 승인과 주문 확정의 순서는 어떻게 정하는가?

주문·재고·결제의 정확한 순서는 업무 정책에 따라 다르다.
재고를 먼저 예약하면 결제 실패 시 예약 해제가 필요하고, 결제를 먼저 승인하면 재고 부족 시 취소 또는 환불이 필요하다.

어느 순서를 선택하든 예약 만료, 보상 처리, 중복 요청, 장애 복구의 상태 전이를 정의해야 한다.
한 번에 모두 성공한다고 가정하지 않고, 부분 성공을 정상 상태로 다룬다.

## 환불과 취소는 어떻게 안전하게 처리하는가?

취소 요청도 중복될 수 있으므로 원 결제 ID와 취소 요청 ID로 멱등성을 보장한다.
부분 취소가 가능하면 누적 취소 금액이 승인 금액을 넘지 않도록 DB에서 조건부 갱신 또는 락으로 검증한다.

PG 취소 성공과 DB 반영이 어긋난 경우를 대비해 조회·재처리·정산 대사 절차를 둔다.
환불 실패를 단순 오류 응답으로 끝내지 않고, 재시도 가능 여부와 운영 확인 대상 상태를 남긴다.

## 금액은 왜 부동소수점으로 계산하면 안 되는가?

`float`, `double`은 이진 부동소수점 표현 오차를 가질 수 있다.
금액은 최소 화폐 단위의 정수(예: 원 단위) 또는 `BigDecimal`로 저장·계산한다.

`BigDecimal`은 생성 시 `new BigDecimal(0.1)` 대신 문자열 또는 `BigDecimal.valueOf`를 사용한다.
나눗셈에는 반올림 정책과 scale을 명시해 금액 규칙이 코드마다 달라지지 않게 한다.

## 정산 대사(Reconciliation)는 왜 필요한가?

서비스 DB 상태와 PG·카드사 정산 데이터는 장애, 지연, 수동 처리 때문에 어긋날 수 있다.
실시간 처리만으로 모든 불일치를 막을 수 있다고 가정하면 안 된다.

주문 ID, 결제 ID, 승인 번호, 금액, 상태, 시각을 기준으로 주기적으로 대사한다.
불일치는 자동 재조회·재처리하거나, 금전 영향이 큰 항목은 별도 검토 상태로 분류한다.

# 네트워크와 외부 연동

## timeout은 왜 연결과 응답 대기를 나눠야 하는가?

연결 timeout은 TCP 연결을 맺지 못할 때까지 기다리는 시간이다.
응답 timeout은 연결된 뒤 서버 응답을 기다리는 시간이다.

둘 중 하나라도 없으면 장애 난 의존성 때문에 요청 스레드와 커넥션 풀이 고갈될 수 있다.
timeout 값은 호출의 중요도, 사용자 대기 시간, 상위 요청의 전체 deadline을 기준으로 정한다.

## 재시도는 언제 위험한가?

재시도는 일시적 네트워크 오류에는 도움이 되지만, 과부하·장애 중인 시스템에는 부하를 증폭시킨다.
쓰기 요청은 중복 실행 위험도 있다.

재시도 대상 오류를 제한하고, 횟수 제한·지수 백오프·jitter·멱등성을 적용한다.
상위 요청 deadline을 넘는 재시도는 하지 않으며, circuit breaker와 함께 사용한다.

## HTTP 상태 코드와 업무 오류는 어떻게 구분하는가?

`400`은 요청 형식 오류, `401`은 인증 필요, `403`은 권한 부족, `404`는 대상 없음에 사용한다.
동시 수정이나 현재 상태와의 충돌은 `409`, 버전 사전 조건 실패는 `412`가 적합할 수 있다.

응답 본문에는 안정적인 오류 코드, 사용자 메시지, trace ID, 필요한 필드 오류를 일관되게 제공한다.
클라이언트가 문자열 메시지에 의존하지 않도록 한다.

## 페이징은 OFFSET과 커서 중 어떻게 선택하는가?

관리 화면처럼 특정 페이지로 바로 이동해야 하면 OFFSET이 단순하다.
다만 뒤 페이지로 갈수록 버릴 행을 많이 읽어야 하고, 중간 변경으로 중복·누락이 생길 수 있다.

최신순 목록과 무한 스크롤에는 커서 기반 페이징이 적합하다.
정렬 기준과 고유 키를 함께 사용해 안정적인 순서를 만들고, 이에 맞는 복합 인덱스를 둔다.

# 테스트와 품질

## 단위 테스트와 통합 테스트의 경계는 어떻게 정하는가?

도메인 규칙과 상태 전이는 단위 테스트로 빠르게 검증한다.
트랜잭션, SQL, 직렬화, 보안 필터, 외부 인프라 연동은 통합 테스트가 필요하다.

Mock은 상호작용 검증에 유용하지만 구현 세부 사항에 과도하게 결합될 수 있다.
핵심 규칙은 가능한 한 상태 기반 테스트를 우선한다.

## Testcontainers는 어디에 쓰는가?

실제 DB, Redis, Kafka를 컨테이너로 실행해 운영 환경과 가까운 통합 테스트를 작성할 수 있다.
인메모리 대체 구현으로 놓치기 쉬운 SQL 문법, 격리 수준, 직렬화, 메시지 동작을 검증하는 데 유용하다.

실행 시간이 길어질 수 있으므로 모든 테스트가 아니라 핵심 경로와 장애 경계에 선별 적용한다.

## 동시성 문제는 어떻게 테스트하는가?

스레드가 같은 시점에 시작하도록 `CountDownLatch`나 barrier를 사용한다.
동시에 요청한 뒤 재고, 잔액, 처리 건수처럼 반드시 지켜야 하는 불변 조건을 검증한다.

한 번 통과했다고 안전하다고 결론 내리지 않는다.
반복 실행, 다양한 스레드 수, DB 격리 수준, timeout과 재시도를 포함해 재현 가능성을 높인다.

## 배포 전 호환성은 어떻게 지키는가?

DB 변경은 하위 호환 가능한 스키마 추가를 먼저 배포한다.
애플리케이션이 새·옛 스키마를 모두 읽을 수 있는 기간을 거친 뒤, 사용하지 않는 컬럼과 인덱스를 제거한다.

이벤트와 API도 소비자가 먼저 배포됐는지, 알 수 없는 필드를 무시할 수 있는지, 롤백 시 읽을 수 있는지를 확인한다.

# 운영과 검증

## 성능 개선은 어떤 지표로 판단하는가?

평균 응답 시간만 보면 일부 매우 느린 요청이 가려진다.
p50, p95, p99와 함께 TPS, 오류율, timeout, 큐 대기 시간, DB 커넥션 대기를 본다.

변경 전후 비교는 다음 조건을 맞춘다.

- 데이터 규모와 데이터 분포
- 동시 사용자 수와 요청 분포
- 워밍업 여부와 캐시 상태
- 측정 기간, 배포 버전, 인프라 사양

TPS만 높아지고 오류율이나 대기 시간이 증가했다면 실제 개선으로 보기 어렵다.

## 장애 원인은 어떤 순서로 좁히는가?

증상만으로 느린 쿼리나 네트워크 문제라고 단정하지 않는다.
장애 시각을 기준으로 다음 정보를 같은 요청 흐름에서 대조한다.

- 요청 ID 또는 trace ID
- 오류율과 지연 시간 그래프
- JVM, 스레드 풀, 커넥션 풀, 큐 지표
- DB 느린 쿼리와 락 대기
- Redis, Kafka, 외부 API의 timeout과 오류

재현 가능한 부하 조건을 만들고, 가설을 지표로 반증하면서 병목을 좁힌다.

## Kubernetes Probe는 어떻게 나누는가?

Liveness Probe는 회복 불가능하게 멈춘 컨테이너를 재시작하기 위한 검사다.
DB가 잠시 느리다는 이유까지 Liveness 실패로 만들면 정상 Pod가 반복 재시작될 수 있다.

Readiness Probe는 트래픽을 받을 준비가 됐는지 판단하며, 실패한 Pod는 Service 엔드포인트에서 제외된다.
기동이 긴 애플리케이션은 Startup Probe로 초기화 동안 Liveness 검사를 유예한다.

`requests`는 스케줄링의 최소 보장 자원이고 `limits`는 최대 사용량이다.
메모리 limit은 JVM heap뿐 아니라 메타스페이스와 native memory까지 포함해 산정한다.
